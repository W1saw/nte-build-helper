# NTE Build Helper — статус

_Обновлено: 10.05.2026 — план зафиксирован, инфраструктура ещё не поднята_

## Цель

Личный AI-помощник по сборкам экипировки персонажей в игре **NTE** (Neverness To Everness). Веб-сайт с UI-конструктором: выбираешь персонажа → собираешь экипировку по слотам → ИИ советует, что улучшить, какие синергии работают, что брать в первую очередь. Ответы строго на русском (как термины внутри игры), без английских слов кроме редких заимствований в скобках.

Аудитория — **только Артём и пара друзей** (личный инструмент, не публичный продукт). Это снимает требования к авторизации/масштабированию/модерации. Magic-link login на whitelist e-mail-ов, чтобы посторонние не жгли API-кредиты.

## Контекст разговора (10.05.2026)

Артём явно: «оно будет ставить английские слова и термины вместо русских, так как в игре есть русский». Главный риск качества — **подмена терминологии**. Источники в интернете — преимущественно англоязычные фан-вики и гайды. Оригинальные русские названия из игры есть только в скринах самого Артёма. Поэтому скрины — это не доп.фича, а **источник истины** для глоссария.

Источники, которые есть:
- Фан-вики и сайты с гайдами (англоязычные, основной массив RAG-данных)
- Скрины из игры самого Артёма (русские названия персонажей, предметов, статов)

Источники, которых **нет** (явно ответил Артём 10.05): YouTube-гайды, Reddit/Discord-обсуждения. На MVP не закладываемся, можно добавить в Этапе 2.

MVP-формат — **конструктор сборки на сайте**. НЕ чат, НЕ анализ скрина текущей сборки (пока). Это даёт наглядность и предсказуемый сценарий: пользователь видит слоты, кликает, получает совет.

## Игровая механика экипировки (важно!)

В NTE экипировка — это **«Консоль»** с двумя сущностями: **Картридж** + **8 Модулей**, которые укладываются как тетрис-пазл внутри картриджа. Это не классические weapon/head/body слоты, как в Genshin/MMO.

**Картридж** — «корпус», даёт сетовый бонус. Имеет 1 главный атрибут + 4 доп.атрибута. Принадлежит одному сету (например «Светлячки и лес»).

**Главный атрибут картриджа — рандомный** (роллится из пула при получении). Например на одном экземпляре «Светлячки и лес» выпало «Бонус к урону анимы +37.5%», на другом могло бы выпасть «Атака +%» или «Защита +%». Тип сета не определяется главным статом картриджа.

**Тип урона сета** (если он есть) — определяется только текстом 2pc/4pc бонусов в нижнем блоке (например «Эпический (2): урон анимы +10%» → сет ассоциируется с анимой). Бонусы фиксированы для каждого сета.

**Модуль** — кусочек определённой формы, который занимает клетки на сетке картриджа.
- **Тип** = количество клеток: тип II (2 клетки), тип III (3 клетки), тип IV (4 клетки). **Тип V в игре отсутствует** (подтверждено 10.05.2026).
- Внутри типа — несколько форм: домино гориз/верт (тип II — 2 формы), I-тромино гориз/верт + 4 L-тромино (тип III — 6 форм), I-тетромино, квадрат 2×2, L-тетромино и пр. (тип IV — 4 формы).
- У каждого модуля: 2 главных атрибута (Атака N + ОЗ M) + 4 доп.атрибута. Иногда есть пассивка (например «+6% Шанс крит. удара за каждый модуль типа II»).

**Сетовый бонус** активируется так: **1 картридж сета X + 4 модуля 4-х конкретных форм**, прописанных в самом картридже. На скрине картриджа есть блок «Улучшение картриджа» с 5 иконками: иконка сета + **4 иконки конкретных форм** (разные между собой), которые нужно набрать среди 8 модулей для активации 4-piece бонуса. 2-piece (Эпический) активируется частичным набором.

Например для «Утраченное сияние» нужны: домино горизонтальное + L-тромино + L-тетромино + I-тромино вертикальное.

**Позиция в сетке на статы и бонусы НЕ влияет.** Игрок укладывает модули руками в самой игре — сайту-помощнику тетрис-пазл решать не нужно. Поэтому MVP-конструктор не рендерит сетку: он просто советует «бери картридж сета X + 4 модуля форм Y₁,Y₂,Y₃,Y₄ + 4 любых модуля под пазл».

```

│                                                            │
│  Server actions / API:                                    │

│    GET  /api/cron/crawl → еженедельный краулер вики       │
│                                                            │
└────────────────┬───────────────────────────────────────────┘
                 │
                 ▼ Supabase JS SDK
┌─── Supabase (eu-central-1) ────────────────────────────────┐
│                                                            │
│  Postgres (с pgvector):                                   │
│    characters, items, sets, glossary, guides, screenshots │
│                                                            │
│  Storage bucket `screenshots/` (приватный)                │
│                                                            │
│  Auth: magic link, whitelist через RLS-политику           │
│                                                            │
└────────────────────────────────────────────────────────────┘

         ▲                                ▲
         │                                │
   Gemini API                       Gemini Vision API
   (Gemini 2.5 Pro,                     (та же модель,
    стриминг советов)                распознавание скринов)
```

**Почему этот стек:**
- Vercel + Supabase — у Артёма уже есть. Лишних расходов на инфраструктуру нет.
- pgvector в Supabase — стандарт для RAG, не нужен отдельный векторный БД.
- Gemini 2.5 Pro (`gemini-2.5-pro`) — баланс цена/качество, vision уже встроен в ту же модель.
- Next.js App Router — server actions для синхронных вызовов, streaming через SSE для длинных ответов ИИ.

## Источники RAG

| URL | Что там | Как краулить |
|---|---|---|
| `https://www.prydwen.gg/neverness-to-everness/` | Тиер-листы, страницы персонажей, гайды по оружию/билдам, мета-обзоры | Sitemap `/sitemap.xml` или crawl с корня по ссылкам внутри `/neverness-to-everness/*` |
| `https://game8.co/games/Neverness-to-Everness` | Wiki-формат: персонажи, билды, синергии, ивенты | Crawl с корня, фильтр по подпути `/games/Neverness-to-Everness/*` |
| `https://mobalytics.gg/neverness-to-everness/characters` | База персонажей с детальными страницами по каждому | Сначала листинг `/characters`, потом проход по карточкам |

Все три — англоязычные → именно для них нужен глоссарий из скринов Артёма для маппинга на русские термины.

**Стратегия краулера на MVP** — не писать кастомную логику под каждый сайт, а использовать generic подход: получить HTML → вытащить main content (через `@mozilla/readability` или `cheerio`-фильтр по семантическим тегам) → нарезать на чанки → embeddings. Если окажется, что для одного из источников generic-подход теряет важные данные (например табличные тиер-листы с prydwen) — добавить кастомный парсер для него.

**robots.txt и rate-limiting** — соблюдать `Crawl-delay`, между запросами не меньше 2 секунд, User-Agent с контактом (`nte-build-helper/0.1 (artem.n-claude@boni-brands.com)`). Раз в неделю — щадящий режим.

## Схема БД (минимум)

Каталог + кэш гайдов. На MVP **не храним** инвентарь Артёма (что у него реально на руках) — только справочник «что бывает в игре». Инвентарь добавим в Этап 4, если конструктор окажется недостаточно полезным без этого.

```sql
-- Персонажи
characters (
  id              uuid pk,
  name_ru         text not null,            -- эталон из скринов
  name_en         text,                     -- из англоязычных гайдов
  role            text,                     -- DPS / support / tank
  anima_type      text,                     -- стихия/тип урона
  base_stats      jsonb,
  avatar_url      text,
  notes           text,
  created_at      timestamptz default now()
)

-- Сеты (Картриджи)
-- damage_type_ru опционально выводится из текста 2pc-бонуса (если он буст. конкретный тип урона).
-- Для сетов с бонусами вида "Атака +10%" — damage_type_ru = NULL.
sets (
  id              uuid pk,
  name_ru         text not null,            -- "Светлячки и лес"
  name_en         text,
  damage_type_ru  text,                     -- "анима", "космос" — выведено из 2pc, или NULL
  bonus_2pc_ru    text,                     -- "урон анимы +10%"
  bonus_4pc_ru    text,                     -- описание легендарного бонуса
  notes           text
)

-- Какие 4 формы модулей нужны для активации 4pc-бонуса сета.
-- Ровно 4 строки на сет (position 1..4), формы могут быть РАЗНЫЕ.
set_required_shapes (
  set_id          uuid fk → sets,
  position        int not null,             -- 1..4
  shape_id        text fk → shapes,
  primary key (set_id, position)
)

-- Картриджи (каталог: какие бывают)
-- Главный стат — РАНДОМНЫЙ ролл из main_stat_pool, НЕ фиксирован для сета.
-- Каждый конкретный экземпляр у игрока имеет свой выпавший главный стат.
cartridges (
  id                  uuid pk,
  set_id              uuid fk → sets,
  name_ru             text not null,
  main_stat_pool      jsonb,                -- какие главные статы могут выпасть (с весами/шансами)
  sub_stat_pool       jsonb,                -- какие доп.статы могут роллиться
  rarity              text,
  notes               text
)

-- Формы модулей (каталог фигур-тетрисов)
shapes (
  id              text pk,                  -- "II_h", "II_v", "III_line_h", "IV_square", ...
  type_roman      text not null,            -- "II", "III", "IV", "V"
  cell_count      int not null,             -- 2, 3, 4, 5
  pattern         jsonb not null,           -- координаты клеток [[0,0],[0,1]] и т.д.
  name_ru         text,                     -- "горизонтальное домино", "квадрат 2×2"
  icon_url        text                      -- из Storage, если загрузим
)

-- Модули (каталог)
modules (
  id                  uuid pk,
  shape_id            text fk → shapes,
  type_roman          text not null,        -- денормализовано из shape для удобства фильтров
  main_stat_options   jsonb,                -- "Атака N + ОЗ M" комбинации
  sub_stat_pool       jsonb,
  has_passive         boolean default false,
  passive_text_ru     text,                 -- "+6% Шанс крит. удара за каждый модуль типа II"
  rarity              text,
  notes               text
)

-- Глоссарий RU↔EN (источник истины — скрины)
glossary (
  id              uuid pk,
  term_ru         text not null,
  term_en         text,
  category        text not null,            -- character, set, cartridge, module, shape, stat, mechanic
  source          text,                     -- screenshot_id или url
  confirmed       boolean default false,    -- true если из скрина
  created_at      timestamptz default now(),
  unique(term_ru, category)
)

-- Куски гайдов с эмбеддингами для RAG
guides (
  id              uuid pk,
  source_url      text not null,
  chunk           text not null,
  embedding       vector(1536),
  lang            text default 'en',
  fetched_at      timestamptz default now()
)

-- Скрины Артёма
screenshots (
  id              uuid pk,
  storage_path    text not null,
  parsed_data     jsonb,
  processed_at    timestamptz,
  notes           text,
  uploaded_by     uuid
)

-- Whitelist пользователей
allowed_users (
  email           text pk,
  added_at        timestamptz default now()
)
```

Индексы:
- `glossary(term_ru)`, `glossary(term_en)` — для быстрого маппинга
- `modules(type_roman)`, `modules(shape_id)` — для фильтров в UI «по форме»
- `cartridges(set_id)` — для группировки по сету
- `guides USING ivfflat (embedding vector_cosine_ops)` — для RAG-поиска

**Сидинг каталога форм (`shapes`)** — небольшой статический dataset, заполнить миграцией 0001 на основе скрина «По форме»:
- Тип II — 2 формы (домино гориз/верт)
- Тип III — 6 форм (I-тромино гориз/верт + 4 L-тромино)
- Тип IV — 4 формы (I-тетромино гориз/верт + квадрат 2×2 + L-тетромино)
- Тип V — отсутствует в игре

Для каждой формы — координаты клеток для рендера (jsonb).

**Сидинг каталога сетов (`sets`)** — 12 сетов, видимых на 10.05.2026:

| Сет | Тип урона (из 2pc) | Заметка |
|---|---|---|
| Диабло | ? | — |
| Багрянец: Две бабочки | ? | — |
| Королевский страж | ? | — |
| Мини-мега приключение | ? | — |
| Светлячки и лес | анима | 2pc: «урон анимы +10%» |
| Уличный боец | ? | — |
| Кредо теней | ? | — |
| Скоростной ёж | ? | — |
| Кровь демона: Проклятие | ? | — |
| Утраченное сияние | космос | 2pc: «урон космоса +10%» |
| Ночная таверна Теи | ? | — |
| Тихое поместье | ? | — |

`damage_type_ru` берётся **из текста 2pc-бонуса** (если бонус буст. тип урона). Для сетов с 2pc вида «Атака +10%» / «ОЗ +10%» поле = NULL. Заполняется по скринам каждого сета через Vision-пайплайн (Этап 2).

Логика:
- При выборе сета → подгружаются 4 требуемые формы из `set_required_shapes`. Каждый из 4 «сетовых» слотов фильтрует каталог модулей по СВОЕЙ форме (слот 1 = форма позиции 1, слот 2 = форма позиции 2, и т.д.).
- Слоты «Свободные» — любая форма, для них алгоритм совета лишь подсказывает «возьми такие-то по статам».
- Tetris-укладку **не** рендерим (договорились — не нужно для советов).
- Кнопка «Получить совет» → `/api/advice` с payload (charId, cartridgeId, moduleIds[]) → стрим ответа в правую панель. Ответ: оценка билда, что улучшить, активные/неактивные синергии, приоритет апгрейдов.

## Пайплайн данных

### A. Скрины → Vision → структура (источник русских терминов)

1. Артём загружает скрин на `/admin/upload` (drag-n-drop).
2. Файл идёт в Supabase Storage bucket `screenshots/`.
3. Server action вызывает Gemini Vision (`gemini-2.5-pro`) с промптом «извлеки все игровые термины с этого скрина: имена персонажей, предметов, статов, описания. Верни JSON».
4. Из JSON-ответа:
   - Новые персонажи → в `characters` (если не было)
   - Новые предметы → в `items`
   - Все термины → в `glossary` с `confirmed=true` и `source=screenshot_id`
5. Артём ревьюит результат на странице скрина, может поправить.

### B. Фан-вики → краулер → RAG

1. Vercel cron раз в неделю (`/api/cron/crawl`) тянет страницы со списка источников.
2. Для каждой страницы — нарезка на чанки (~500 токенов с overlap 50).
3. Embeddings через OpenAI `text-embedding-3-small` (или Voyage AI) — 1536 размерностей.
4. Insert в `guides`. При повторном запуске — upsert по `source_url + chunk_index`.
5. Список источников — в env-переменной `CRAWL_SOURCES` (csv URL-ов), чтобы менять без деплоя.

### C. Запрос совета

1. UI собирает payload: `{ characterId, cartridgeId, moduleIds: [m1, m2, ..., m8] }`.
2. `/api/advice` тянет из БД полный контекст: характеристики персонажа, выбранный картридж + сет с бонусами, 8 модулей с их формами/статами, проверка активны ли 2pc/4pc.
3. Делает векторный поиск в `guides` по запросу `{character.name_en} build guide` — берёт top-5 чанков.
4. Грузит весь `glossary` (он маленький — сотни строк).

## Решения по дороге

### Решение: Next.js + Vercel + Supabase, не отдельный backend (10.05.2026)

Артём спросил про Telegram-бот ИЛИ сайт. Выбрал **сайт первым** потому что: (а) в браузере проще дебажить промпты и видеть полные ответы; (б) UI конструктора — ключевая фича, в Telegram её собирать неудобно; (в) бот можно добавить вторым интерфейсом к тому же API позже без переделки. Стек Vercel+Supabase у Артёма уже есть — никаких новых платных сервисов.

### Решение: Gemini 2.5 Pro (один LLM на советы и Vision) (10.05.2026)

Gemini 2.5 Pro умеет в vision из коробки — не нужен отдельный OCR (Tesseract или Google Vision). Один API, один билиннг, один промпт-стиль. Если окажется, что Vision на скринах с мелким шрифтом теряет точность — можно подключить отдельный OCR как fallback.

### Решение: pgvector в Supabase, не отдельный векторный БД (10.05.2026)

Pinecone/Weaviate/Qdrant — всё это лишние сервисы. Supabase Postgres с pgvector хватает для тысяч чанков (для личного проекта — за глаза). При запросе делается один SQL-join `guides ⊳ items ⊳ characters`, не нужно синхронизировать две БД.

### Решение: глоссарий из скринов как «эталон» русских терминов (10.05.2026)

Главная боль — англоязычная терминология в ответах. Решено: скрины Артёма → Gemini Vision → русские термины с пометкой `confirmed=true`. Эти термины приоритетнее любых вики. В системный промпт инжектируется весь `glossary` (он маленький). Бот, встретив незнакомый термин, ставит `[?]` — это становится фидбек-петлёй: Артём по этим меткам понимает, какой скрин нужно загрузить следующим.

### Решение: магик-линк на whitelist, не публичный (10.05.2026)

Артём явно: «только я / пара друзей». Нет смысла строить полноценную регистрацию/модерацию — просто таблица `allowed_users` с мейлами + RLS-политика, что писать в БД могут только они. Любой публичный посетитель максимум увидит landing, не сможет жечь API.

### Решение: MVP БЕЗ анализа скрина текущей сборки (10.05.2026)

Артём ответил MVP = конструктор. Идея «кидаешь скрин текущей сборки → ИИ говорит что улучшить» отложена в Этап 4. Причина: конструктор требует только списка предметов в БД, а анализ скрина — ещё и pixel-perfect распознавание UI игры (риск ошибок). Сначала довести базу, потом расширять.

### Решение: НЕ закладываем YouTube/Reddit на MVP (10.05.2026)

Артём ответил, что есть только сайты с гайдами + свои скрины. YouTube-транскрипты через Whisper и парсинг Reddit — отложены в Этап 4. Если фан-вики окажутся слишком скудны для качественных советов — вернёмся.

### Решение: НЕ рендерим Tetris-сетку на MVP (10.05.2026)

После уточнения механики (8 модулей разной формы укладываются как тетрис в картридж): позиция в сетке на статы и бонусы НЕ влияет — это чистый spatial-пазл. Артём решает его руками в самой игре. Сайт-помощник советует «бери картридж сета X + 4 модуля формы Y + 4 любых». Если позже окажется что определённые комбинации форм невозможно уложить — добавим валидатор раскладки в Этап 4.

### Решение: схема каталог + кэш гайдов, БЕЗ инвентаря Артёма на MVP (10.05.2026)

В БД храним «что бывает в игре» (каталог персонажей/картриджей/модулей/форм/сетов) — чтобы конструктор предлагал из общего пула. НЕ храним «что есть у Артёма на руках» — это потребовало бы регулярной синхронизации с игрой и сильно усложнило бы UI. Если выяснится, что без инвентаря советы непрактичные («ты предложил картридж +20 которого у меня нет») — добавим в Этап 4 загрузку инвентаря через скрины.

### Решение: главный стат картриджа = рандомный ролл, не свойство сета (10.05.2026)

Артём явно поправил: верхний главный стат картриджа («Бонус к урону анимы 37.5%») — рандомный ролл из пула, а не свойство сета. У сета фиксированы только тексты 2pc/4pc-бонусов в нижнем блоке. Поэтому в схеме `cartridges.main_stat_pool` (jsonb список возможных значений с весами), а `sets.damage_type_ru` выводится только из текста 2pc-бонуса (и может быть NULL для сетов без буста типа урона).
